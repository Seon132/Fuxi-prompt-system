# 四层架构动态熵机制（V2 可执行评分模板）

## 0. 这份模板是干什么的

这不是把 V2 里的所有数学公式都逐项硬算一遍。

它的用途是：

1. 把 V2 里最重要的公式，压缩成可以人工打分或半自动打分的表。
2. 让不懂数学的人，也能先把这套机制跑起来。
3. 先形成一个 **V2 代理执行版**，以后再逐步升级成更强的自动计算版。

因此，本模板采用三层结构：

- `定义层`：解释系统里到底有哪些数学对象
- `评分层`：真正用于打分的项目
- `执行层`：每次设计提示词或审核输出时，实际要走的步骤

---

## 1. 先说结论：不是每个公式都要执行

V2 里的公式可以分成三类。

### 1.1 A 类：定义公式

这类公式主要用来说明“系统是什么”，不需要每次都手算。

包括：

- `G_t = (V_t, A_t, B_t, C_t, x_t, M_t)`
- `A_t`
- `B_t`
- `C_t`
- `H^*`

这些公式的作用是：

- 帮你把系统对象想清楚
- 帮你知道应该记录哪些信息
- 帮你知道评分项目从哪里来

它们像“设计图纸”，不是每次都拿计算器去算。

### 1.2 B 类：控制公式

这类公式说明系统理论上应该怎样运作，但在 V2 初期通常只做“代理执行”。

包括：

- `ER(S_t,t)`
- `Syn(S_t,t)`
- `Act(S,t)`
- `J_t(Y)`

这类公式在现阶段的落地方式通常不是精确数学求值，而是：

- 先转成打分项
- 再按权重合成总分

### 1.3 C 类：真正执行公式

这类公式在实际使用时会真的参与打分。

包括：

- `Score_static`
- `Score_runtime`
- 重排阈值

所以最重要的一句话是：

**V2 不是“每个公式每次都算”，而是“用少量总分公式，调用一组代理评分项”。**

---

## 2. 落地时分成两个环节

### 2.1 环节一：提示词设计阶段

这个阶段不看真实对话，只看系统提示词本身写得怎么样。

这个阶段真正执行的是：

- `Score_static`

它回答的是：

- 四层是否齐全
- 层间连接是否清楚
- 层内是否有主次结构
- 是否定义了组合协同/冲突
- 是否定义了审核、重排、回写规则

### 2.2 环节二：提示词运行阶段

这个阶段看某一轮真实输入输出是否运行得合理。

这个阶段真正执行的是：

- `Score_runtime`
- `J_t`
- 重排触发规则

它回答的是：

- 这次输出有没有按正确的路径收敛
- 有没有证据、边界、反例
- 当前组合调用是否合理
- 低分时是否要重排

---

## 3. 实际落地时的最简工作流

建议先只跑下面这 6 步。

### 3.1 设计阶段

1. 先写系统提示词。
2. 用静态评分表打一次 `Score_static`。
3. 若低于 70，先不部署，先修改结构。

### 3.2 运行阶段

1. 用户输入一个问题。
2. 模型给出候选回答。
3. 用动态评分表打一次 `Score_runtime`。
4. 再用审核表判断 `J_t` 是否达标。
5. 若低于阈值，则重排。
6. 达标后再输出。

所以实际执行时，常用的是：

- 一张静态表
- 一张动态表
- 一张重排判断表

## 3.5 固定哲学预设说明

哲学层分为两个层级：

1. 固定原原则层：系统最高约束层，参与基准评分。
2. 扩展原则层：角色/任务特定哲学原则层，不参与基准评分。

### 3.5.1 固定原原则层

固定哲学原理集：

- 本质主义
- 第一性原理
- 体系性思维
- 辩证法
- 矛盾论
- 具体问题具体分析
- 实证主义
- 实用主义

强制元约束：

- 体系性思维闭环

固定闭环路径：

`经验 -> 知识 -> 思维 -> 哲学 -> 思维 -> 知识 -> 经验`

这组固定原原则的地位是：

1. 不可删除。
2. 不可替换。
3. 不可降级。
4. 始终高于任何角色原则、任务原则与临时策略。

其中，`体系性思维`及其闭环是系统最高原原则，承担全局治理职责：

- 边界收敛
- 结构约束
- 冲突仲裁
- 版本稳态

### 3.5.2 扩展原则层

在固定原原则层之下，允许生成角色特定、任务特定或领域特定的扩展哲学原则。

扩展原则的使用规则：

1. 扩展原则只能在固定原原则之下生长，不能脱离其约束。
2. 扩展原则可以补充角色风格、领域偏好、特殊方法，但不能替代固定原原则。
3. 扩展原则不计入哲学基准评分，不因“多”而加分。
4. 若扩展原则与固定原原则冲突，按“固定原原则优先”处理，并视为结构违宪。
5. 若扩展原则破坏体系性思维闭环，直接判为高严重度缺陷。

### 3.5.3 评分规则总说明

所有静态评分与动态评分，都以固定原原则层作为统一判准。

扩展原则只做两件事：

1. 记录其是否存在、是否清晰、是否适配角色目标。
2. 检查其是否服从固定原原则与体系性思维闭环。

- 每个子项都按 `0.0 ~ 1.0` 打分

### 4.2.1 哲学预设专门评分

建议按以下四项先打子分，再合成 `S_philosophy_constitution`：

`S_philosophy_constitution = 0.25 P_coverage + 0.25 P_binding + 0.25 P_loop_static + 0.25 P_definition`

其中：

- `P_coverage`：8 个固定原理是否完整保留
- `P_binding`：是否明确为不可删除、不可替换、不可降级的系统预设，且扩展原则不得越位
- `P_loop_static`：是否明确定义体系性思维闭环及其职责，并把它设为最高原原则
- `P_definition`：各固定原理是否给出清晰的架构作用

补充规则：

1. 扩展原则本身不单独计分。
2. 但若扩展原则试图替代、覆盖、削弱固定原原则，则必须拉低 `P_binding` 或 `P_loop_static`。
3. 若扩展原则只是角色化补充，且服从固定原原则，则不加分也不扣分，可在报告中单独记录。

## 5. 动态评分表（运行阶段）

### 5.1 评分公式

`Score_runtime = 100 * (0.12 S_order_dyn + 0.12 S_edge_dyn + 0.16 S_combo_dyn + 0.18 S_valid + 0.10 S_control + 0.10 S_writeback + 0.10 S_revise + 0.12 S_philosophy_enforce)`

### 5.2 评分表

| 项目 | 代号 | 权重 | 怎么看 | 评分标准（0~1） |
|---|---:|---:|---|---|
| 是否体现正确降熵路径 | `S_order_dyn` | 0.12 | 是否先理解输入，再组织知识，再形成主线，再用原则校验 | 混乱=0~0.3；基本合理=0.4~0.7；清晰稳定=0.8~1.0 |
| 实际层间作用是否合理 | `S_edge_dyn` | 0.12 | 输出中是否能看到层间调用关系是对的 | 很弱=0~0.3；一般=0.4~0.7；明显合理=0.8~1.0 |
| 组合调用是否合理 | `S_combo_dyn` | 0.16 | 当前激活的节点组合是否互补，而不是互相打架 | 冲突多=0~0.3；一般=0.4~0.7；协同明显=0.8~1.0 |
| 证据/边界/反例是否完整 | `S_valid` | 0.18 | 是否交代根据、条件、限制、反例 | 缺很多=0~0.3；一般=0.4~0.7；完整=0.8~1.0 |
| 外部控制是否真的生效 | `S_control` | 0.10 | 回答是否明显经过规则约束，而不是自由发挥 | 几乎无=0~0.3；有一点=0.4~0.7；很明显=0.8~1.0 |
| 是否可回写 | `S_writeback` | 0.10 | 本轮结果是否能沉淀为知识、经验、规则更新 | 不可回写=0~0.3；部分可回写=0.4~0.7；可直接沉淀=0.8~1.0 |
| 低分时是否修订 | `S_revise` | 0.10 | 若有缺陷，系统是否自动补证据/重排/修正 | 没修订=0~0.3；有一点=0.4~0.7；修订有效=0.8~1.0 |
| 固定哲学预设是否真正介入 | `S_philosophy_enforce` | 0.12 | 是否体现固定 8 原原则与体系性闭环对本轮输出的真实约束；若调用扩展原则，是否仍受固定原原则门控 | 未介入=0~0.2；弱介入=0.3~0.5；明显介入且可追踪=0.6~0.8；强制门控并触发修订=0.81~1.0 |

### 5.2.1 哲学预设动态评分

建议按以下五项先打子分，再合成 `S_philosophy_enforce`：

`S_philosophy_enforce = 0.30 P_loop_trace + 0.25 P_boundary_gate + 0.20 P_conflict_arbitration + 0.15 P_evidence_enforce + 0.10 P_writeback_enforce`

其中：

- `P_loop_trace`：输出是否能回溯到固定闭环链路
- `P_boundary_gate`：是否体现边界收敛与结构约束
- `P_conflict_arbitration`：是否体现主要/次要矛盾识别与仲裁
- `P_evidence_enforce`：是否体现实证主义式证据门控
- `P_writeback_enforce`：是否体现版本稳态与回写

补充规则：

1. 扩展原则不单独加分。
2. 若本轮输出调用扩展原则，只检查其是否服从固定原原则与体系性思维闭环。
3. 若扩展原则造成越界、冲突或替代固定原原则，必须拉低 `S_philosophy_enforce`，必要时直接触发重排。

### 5.2.2 运行期增强规则

为提高动态可审计性，建议把以下三条作为通用运行期规则：

1. 输出前必须完成 `证据 / 边界 / 反例 / 适用范围 / 低置信部分` 检查。
2. 这组检查可以外显写入最终回答，也可以作为隐式审核块存在，但评分时都必须视为有效审查对象。
3. 若调用扩展原则，必须记录其受哪一条或哪几条固定原原则门控。
4. 若无法说明扩展原则的上位门控关系，则不得给高 `S_philosophy_enforce` 分。
5. 若涉及来源敏感判断而无文本依据或可靠证据，必须降级为 `under_review`。

这里所说的来源敏感判断包括但不限于：

- 作者立场归属
- 文本出处归属
- 思想史定位
- 具体史实断言
- 直接批评关系或学术影响链判断

评分联动规则：

1. 缺少 `证据 / 边界 / 反例 / 适用范围 / 低置信部分` 检查时，优先拉低 `S_valid` 与 `S_control`。
2. 调用扩展原则但未说明受哪条固定原原则门控时，优先拉低 `S_philosophy_enforce`。
3. 对来源敏感判断无依据却未降级为 `under_review` 时，优先拉低 `S_valid`，并提高 `RuleViolation`。

## 6. 审核目标函数的数学落地表

运行阶段直接执行：

`J_t(Y) = eta_1 Score_runtime + eta_2 Consistency + eta_3 EvidenceCov + eta_4 GoalFit - eta_5 EntropyExcess - eta_6 RuleViolation - eta_7 UnsafeRisk`

其中建议先把所有输入都标准化到 `0~1` 区间。

### 6.1 直接计算表

建议初始权重：

- `eta_1 = 0.30`
- `eta_2 = 0.15`
- `eta_3 = 0.15`
- `eta_4 = 0.15`
- `eta_5 = 0.10`
- `eta_6 = 0.10`
- `eta_7 = 0.05`

则可直接计算：

`J_t(Y) = 0.30 Score_runtime + 0.15 Consistency + 0.15 EvidenceCov + 0.15 GoalFit - 0.10 EntropyExcess - 0.10 RuleViolation - 0.05 UnsafeRisk`

### 6.2 审核表

| 项目 | 含义 | 分值范围 |
|---|---|---:|
| `Score_runtime` | 动态结构总分折算到 0~1 | 0~1 |
| `Consistency` | 回答内部是否一致 | 0~1 |
| `EvidenceCov` | 证据覆盖度是否够 | 0~1 |
| `GoalFit` | 是否对准用户问题 | 0~1 |
| `EntropyExcess` | 是否不必要地发散 | 0~1（越高越差） |
| `RuleViolation` | 是否违反规则 | 0~1（越高越差） |
| `UnsafeRisk` | 是否有风险或越界 | 0~1（越高越差） |

### 6.3 计算规则

请按下面方式直接计算：

1. 先计算 `Score_runtime / 100`，得到 `0~1` 的结构质量值。
2. 再由 AI 审核器分别给出：
   - `Consistency`
   - `EvidenceCov`
   - `GoalFit`
   - `EntropyExcess`
   - `RuleViolation`
   - `UnsafeRisk`
3. 把上述值代入 `J_t(Y)`。
4. 得到最终审核值 `J_t`。

补充执行规则：

1. `EvidenceCov` 的判断必须特别检查：是否完成 `证据 / 边界 / 反例 / 适用范围 / 低置信部分` 审核。
2. 若存在来源敏感判断，必须额外检查：是否给出文本依据、可靠来源，或明确降级为 `under_review`。
3. 若调用扩展原则，必须额外检查：是否说明其受哪条固定原原则门控。

### 6.4 判定阈值

建议阈值：

- `J_t >= 0.75`：通过，可输出
- `0.60 <= J_t < 0.75`：有条件通过，建议修订后输出
- `J_t < 0.60`：不通过，触发重排

### 6.5 强制拦截条件

即使 `J_t` 达标，只要满足任一条件，也必须进入重排：

1. `RuleViolation >= 0.30`
2. `UnsafeRisk >= 0.25`
3. `EvidenceCov < 0.50`
4. `GoalFit < 0.60`
5. 存在来源敏感判断且无依据、无降级，视为严重审核缺口，应提高 `RuleViolation` 并优先进入重排

## 7. “集合熵效应”怎么数学化落地

V2 里有：

- `ER(S_t,t)`
- `Syn(S_t,t)`

这里不再只做口语化判断，而是定义成 AI 可执行的代理计算。

### 7.1 联合降熵贡献 `ER(S_t,t)`

定义：

`ER(S_t,t) = H_t - H_t^(S_t)`

在工程落地里，先不直接计算真实文本熵，而是令：

`ER_proxy(S_t,t) = 0.35 MainlineGain + 0.25 AmbiguityReduction + 0.20 StructureGain + 0.20 DecisionCompression`

其中各项都标准化到 `0~1`：

- `MainlineGain`：是否更清楚地主线化
- `AmbiguityReduction`：是否减少歧义
- `StructureGain`：是否增加结构清晰度
- `DecisionCompression`：是否把可行路径压缩到更合理的范围

若无真实熵估计器，则令：

`ER(S_t,t) approx ER_proxy(S_t,t)`

### 7.2 组合协同 `Syn(S_t,t)`

定义：

`Syn(S_t,t) = ER(S_t,t) - sum_(n in S_t) ER(n,t)`

工程代理计算：

`Syn_proxy(S_t,t) = 0.40 Complementarity + 0.25 NonRedundancy + 0.20 ConstraintSupport - 0.15 ConflictCost`

其中：

- `Complementarity`：不同节点是否提供互补作用
- `NonRedundancy`：是否避免机械重复
- `ConstraintSupport`：组合后是否更符合哲学/规则约束
- `ConflictCost`：是否出现互相抵消或打架

若暂不精算，则令：

`Syn(S_t,t) approx Syn_proxy(S_t,t)`

## 8. 最推荐的落地方式：从阶段二开始

### 8.1 当前推荐起点：阶段二半自动评分

适合作为现在的默认模式。

做法：

- 由 AI 自动完成所有子项评分
- 由 AI 自动计算 `Score_static`、`Score_runtime`、`J_t`
- 由 AI 自动生成“优点/问题/建议”评估报告
- 人只负责查看结果，必要时决定是否采用建议

你基本不需要手工计算。

人的主要参与点只有三类：

- 确认评分权重是否要调整
- 确认某些高风险任务是否允许自动通过
- 查看 AI 给出的修改建议是否符合你的目标

### 8.2 阶段三：全自动评分

适合后续接入程序化检查器。

做法：

- 把评分项写成结构化提示或代码检查器
- 自动提取证据、反例、边界、层间调用痕迹
- 自动计算所有代理子分项
- 自动计算 `J_t`
- 自动触发重排与再生成

### 8.3 为什么现在不直接从“人工版”开始

因为你的目标本来就是让智能体自己审核自己。

所以从执行方式上，最合理的默认就是：

**AI 计算，AI 解释，AI 给建议，人做监督。**

## 9. 最小执行版本：你现在真正需要用的只有这些

如果你现在就要用，不需要管所有公式。

你只要先抓住这 5 个东西：

1. `Score_static`
2. `Score_runtime`
3. `S_combo_dyn`
4. `S_valid`
5. 重排阈值

也就是说：

- 设计阶段主要看结构是否完整
- 运行阶段主要看组合是否合理、证据是否完整、是否需要重排

## 10. 每次使用时的执行顺序

建议每次按这个顺序由 AI 自动执行。

### 10.1 设计阶段填写顺序

1. 看四层是否清楚
2. 看层间连接是否清楚
3. 看层内主次是否清楚
4. 看是否定义组合机制
5. 看是否定义审核与回写
6. 算 `Score_static`

### 10.2 运行阶段填写顺序

1. 看输出是否有主线
2. 看层间调用是否合理
3. 看节点组合是否协同
4. 看证据/边界/反例是否完整
5. 看外部规则是否真的介入
6. 看是否完成 `证据 / 边界 / 反例 / 适用范围 / 低置信部分` 审核
7. 若调用扩展原则，记录其受哪条固定原原则门控
8. 若存在来源敏感判断且无依据，降级为 `under_review`
9. 看是否需要修订
10. 算 `Score_runtime`
11. 决定是否重排

## 11. 评分之后必须输出评估报告

无论是静态评级还是动态评级，都不能只给分数。

每次评分后，AI 必须额外输出一份结构化评估报告。

### 11.1 静态评估报告格式

| 模块 | 内容要求 |
|---|---|
| 总分与评级 | 输出 `Score_static` 和评级 |
| 通过/不通过结论 | 明确说明是否建议部署 |
| 扩展原则记录 | 若存在扩展原则，记录其名称、作用与是否服从固定原原则 |
| 做得好的地方 | 至少列出 2-5 条优势 |
| 一般的地方 | 至少列出 1-3 条中等表现项 |
| 最薄弱的地方 | 至少列出 1-3 条薄弱项 |
| 结构性建议 | 给出对应修改建议，明确改哪一层、哪一条边、哪一类规则 |

### 11.2 动态评估报告格式

| 模块 | 内容要求 |
|---|---|
| 总分与评级 | 输出 `Score_runtime`、`J_t` 和评级 |
| 是否通过 | 明确说明“直接输出 / 修订后输出 / 重排” |
| 扩展原则记录 | 若本轮调用扩展原则，记录其是否受固定原原则门控 |
| 审核记录 | 记录 `证据 / 边界 / 反例 / 适用范围 / 低置信部分` 是否完成 |
| 来源敏感判断记录 | 若存在来源敏感判断，记录其依据情况，或是否已降级为 `under_review` |
| 做得好的地方 | 指出本轮哪些层、哪些组合、哪些证据环节做得好 |
| 一般的地方 | 指出哪些地方勉强合格但不稳定 |
| 最薄弱的地方 | 指出最拖分的 1-3 个子项 |
| 重排建议 | 若不通过，明确下一轮该补什么、删什么、加强什么 |

### 11.3 建议输出模版

```text
[评估结论]
- 总分：
- 评级：
- 是否通过：
- 扩展原则记录：
- 审核记录：
- 来源敏感判断记录：

[做得好的地方]
- 1.
- 2.

[一般的地方]
- 1.
- 2.

[最薄弱的地方]
- 1.
- 2.

[修改建议]
- 1.
- 2.
- 3.
```

这一步不是附属项，而是评分系统必须产出的正式结果。

## 12. 一句话说明“公式怎么用”

最简单的理解方式是：

- 大公式用来定义系统结构
- 中层公式用来支持 AI 计算子分项
- 总分公式 `Score_static`、`Score_runtime`、`J_t` 负责真正执行
- 评估报告负责把结果翻译成可修改的建议
